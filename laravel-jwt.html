01 · Concept

What is a JWT, really?

A JSON Web Token is a compact, self-contained way of proving who a user is, without the server having to remember anything about them between requests. Look at the diagram above — a token is just three pieces glued together with dots:

  • Header — says which signing algorithm was used.
  • Payload — holds the actual data, like the user's ID (sub) and an expiry time (exp).
  • Signature — a cryptographic seal that proves nobody tampered with the first two parts.

The login flow works like this: the user sends their email and password once, the server checks them and hands back a token, and from then on the client just attaches that token to every request instead of logging in again.

Why this matters

Because the signature can only be created by someone who knows the server's secret key, nobody can edit a token (say, changing their user ID to someone else's) without the server immediately noticing that the signature no longer matches.

02 · Setup

Starting a fresh Laravel project

If you already have a project, you can skip straight to the next step. Otherwise, this creates a new Laravel app and gets it ready to connect to a database.

terminal
# create a new Laravel project
composer create-project laravel/laravel jwt-demo
cd jwt-demo
What's happening here

composer create-project downloads Laravel and all its core dependencies into a new folder called jwt-demo. The second line just moves your terminal into that new folder so the following commands run in the right place.

Then point Laravel at your database by editing the .env file in the project root:

.env
DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=jwt_demo
DB_USERNAME=root
DB_PASSWORD=
What's happening here

The .env file holds environment-specific settings that shouldn't be hard-coded into your app — database credentials, secret keys, and so on. Laravel reads these values automatically when it boots.

03 · Install

Adding the JWT package

Laravel doesn't include JWT support by default, so we bring in the most widely used community package for it, tymon/jwt-auth.

terminal
composer require tymon/jwt-auth
What's happening here

This downloads the package and wires it into Laravel via Composer, PHP's dependency manager — the same tool you used to create the project itself.

terminal
php artisan vendor:publish --provider="Tymon\JWTAuth\Providers\LaravelServiceProvider"
What's happening here

This copies the package's default settings file into your project as config/jwt.php, where you can later tweak things like how long a token stays valid.

terminal
php artisan jwt:secret
What's happening here

This generates a random secret key and saves it as JWT_SECRET in your .env file. It's the key used to sign every token, so treat it like a password — never commit it to a public repository.

04 · Model

Preparing the User model

JWT needs your User model to implement a contract called JWTSubject, so the package knows how to attach a user's identity to a token.

app/Models/User.php
<?php

namespace App\Models;

use Illuminate\Foundation\Auth\User as Authenticatable;
use Illuminate\Notifications\Notifiable;
use Tymon\JWTAuth\Contracts\JWTSubject;

class User extends Authenticatable implements JWTSubject
{
    use Notifiable;

    protected $fillable = [
        'name',
        'email',
        'password',
    ];

    protected $hidden = [
        'password',
        'remember_token',
    ];

    // tells JWT which field identifies the user
    public function getJWTIdentifier()
    {
        return $this->getKey();
    }

    // lets you attach extra data to the token if needed
    public function getJWTCustomClaims()
    {
        return [];
    }
}
What's happening here
  • implements JWTSubject is what marks this model as "JWT-aware."
  • getJWTIdentifier() returns the value that will go in the token's sub claim — here, the user's primary key (their ID).
  • getJWTCustomClaims() lets you stuff extra information into the token payload, like a role or tenant ID. We're leaving it empty for now — that's perfectly normal for a basic setup.
05 · Guard

Configuring the auth guard

Laravel decides how to authenticate a request using something called a "guard." We need to tell the api guard to use JWT instead of the default session-based driver.

config/auth.php
'defaults' => [
    'guard' => 'api',
    'passwords' => 'users',
],

'guards' => [
    'web' => [
        'driver' => 'session',
        'provider' => 'users',
    ],

    'api' => [
        'driver' => 'jwt',
        'provider' => 'users',
    ],
],
What's happening here

Setting 'driver' => 'jwt' on the api guard tells Laravel: "when something is protected by this guard, check for a JWT instead of a session cookie." Making api the default guard means routes using plain auth middleware will use JWT automatically.

06 · Controller

Building the auth controller

This is the heart of the system — a controller that handles registering, logging in, checking who's logged in, refreshing tokens, and logging out.

terminal
php artisan make:controller AuthController
What's happening here

This is an Artisan command (Laravel's built-in CLI tool) that scaffolds a blank controller file for you at app/Http/Controllers/AuthController.php.

app/Http/Controllers/AuthController.php
<?php

namespace App\Http\Controllers;

use App\Models\User;
use Illuminate\Http\Request;
use Illuminate\Support\Facades\Hash;
use Illuminate\Support\Facades\Validator;
use Tymon\JWTAuth\Facades\JWTAuth;

class AuthController extends Controller
{
    public function register(Request $request)
    {
        $validator = Validator::make($request->all(), [
            'name'     => 'required|string|max:255',
            'email'    => 'required|string|email|max:255|unique:users',
            'password' => 'required|string|min:6|confirmed',
        ]);

        if ($validator->fails()) {
            return response()->json($validator->errors(), 422);
        }

        $user = User::create([
            'name'     => $request->name,
            'email'    => $request->email,
            'password' => Hash::make($request->password),
        ]);

        $token = JWTAuth::fromUser($user);

        return response()->json([
            'message'      => 'User registered successfully',
            'user'         => $user,
            'access_token' => $token,
            'token_type'   => 'bearer',
        ], 201);
    }

    public function login(Request $request)
    {
        $credentials = $request->only('email', 'password');

        $validator = Validator::make($credentials, [
            'email'    => 'required|email',
            'password' => 'required|string',
        ]);

        if ($validator->fails()) {
            return response()->json($validator->errors(), 422);
        }

        if (! $token = JWTAuth::attempt($credentials)) {
            return response()->json(['error' => 'Invalid email or password'], 401);
        }

        return $this->respondWithToken($token);
    }

    public function me()
    {
        return response()->json(auth()->user());
    }

    public function logout()
    {
        auth()->logout();

        return response()->json(['message' => 'Successfully logged out']);
    }

    public function refresh()
    {
        return $this->respondWithToken(auth()->refresh());
    }

    protected function respondWithToken($token)
    {
        return response()->json([
            'access_token' => $token,
            'token_type'   => 'bearer',
            'expires_in'   => auth()->factory()->getTTL() * 60,
        ]);
    }
}
What's happening here, method by method
  • register() — validates the incoming data, hashes the password (never store plain-text passwords), creates the user, then immediately issues them a token with JWTAuth::fromUser() so they don't have to log in separately right after signing up.
  • login() — checks the email and password against the database using JWTAuth::attempt(). If they match, it returns a signed token; if not, a 401 error.
  • me() — a simple endpoint that returns whichever user the current token belongs to. Handy for letting a frontend fetch "who am I logged in as."
  • logout() — invalidates the current token so it can't be reused.
  • refresh() — issues a brand-new token and invalidates the old one, so a user can stay logged in without re-entering their password.
  • respondWithToken() — a small shared helper so every endpoint that returns a token formats its response the same way, including expires_in, calculated from the TTL (time-to-live) setting in config/jwt.php.
07 · Routes

Defining the routes

Routes are what connect a URL and HTTP verb to a controller method. These go in routes/api.php.

routes/api.php
<?php

use App\Http\Controllers\AuthController;
use Illuminate\Support\Facades\Route;

Route::group([
    'prefix' => 'auth',
], function ($router) {
    Route::post('register', [AuthController::class, 'register']);
    Route::post('login', [AuthController::class, 'login']);
    Route::post('logout', [AuthController::class, 'logout']);
    Route::post('refresh', [AuthController::class, 'refresh']);
    Route::get('me', [AuthController::class, 'me']);
});
What's happening here

Route::group(['prefix' => 'auth'], ...) means every route inside automatically starts with /auth. Since Laravel already prefixes everything in api.php with /api, the final endpoints look like this:

Method & PathPurpose
POST /api/auth/registerCreate a new account
POST /api/auth/loginLog in and receive a token
POST /api/auth/logoutInvalidate the current token
POST /api/auth/refreshExchange an old token for a new one
GET /api/auth/meGet the currently logged-in user
08 · Middleware

Protecting routes

Right now, all five routes are wide open — anyone can call me or logout without a token. Middleware is how we lock that down.

routes/api.php
Route::group([
    'prefix' => 'auth',
], function ($router) {
    Route::post('register', [AuthController::class, 'register']);
    Route::post('login', [AuthController::class, 'login']);

    Route::group(['middleware' => 'auth:api'], function ($router) {
        Route::post('logout', [AuthController::class, 'logout']);
        Route::post('refresh', [AuthController::class, 'refresh']);
        Route::get('me', [AuthController::class, 'me']);
    });
});
What's happening here

The inner Route::group(['middleware' => 'auth:api'], ...) wraps just the routes that should require a valid token. Middleware runs before the controller does — if the request has no token, or an invalid one, Laravel stops it and automatically returns a 401 Unauthorized response before your controller code even runs.

09 · Testing

Trying it out with curl

You can test any of these endpoints with Postman, Insomnia, or plain curl in your terminal.

terminal — register
curl -X POST http://localhost:8000/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Jane Doe",
    "email": "jane@example.com",
    "password": "secret123",
    "password_confirmation": "secret123"
  }'
What's happening here

This sends a JSON body to the register endpoint. The response includes an access_token — copy that value, you'll need it for the next requests.

terminal — protected route
curl -X GET http://localhost:8000/api/auth/me \
  -H "Authorization: Bearer YOUR_TOKEN_HERE"
What's happening here

The Authorization: Bearer <token> header is how the client proves who it is on every request. If the token is valid, Laravel returns Jane's user details; if it's missing or wrong, you get a 401 instead.

10 · Errors

Handling expired tokens gracefully

By default, tokens expire after 60 minutes. When that happens, JWTAuth throws an exception — it's worth catching these so your API returns clean JSON instead of a raw error page.

bootstrap/app.php (Laravel 11+)
use Tymon\JWTAuth\Exceptions\TokenExpiredException;
use Tymon\JWTAuth\Exceptions\TokenInvalidException;
use Tymon\JWTAuth\Exceptions\JWTException;

->withExceptions(function ($exceptions) {
    $exceptions->render(function (TokenExpiredException $e, $request) {
        return response()->json(['error' => 'Token has expired'], 401);
    });

    $exceptions->render(function (TokenInvalidException $e, $request) {
        return response()->json(['error' => 'Token is invalid'], 401);
    });

    $exceptions->render(function (JWTException $e, $request) {
        return response()->json(['error' => 'Token is missing'], 401);
    });
});
What's happening here

Each $exceptions->render() call tells Laravel: "if this specific type of error occurs anywhere in the app, respond with this JSON instead of the default error page." This keeps every error your API sends back consistent and easy for a frontend to handle.

Note for older projects

On Laravel 10 or earlier, this same logic goes inside the render() method of app/Exceptions/Handler.php instead of bootstrap/app.php.

11 · Pitfalls

Common beginner mistakes

  • Skipping php artisan jwt:secret — without it, every token fails to sign and you'll get confusing errors.
  • Storing tokens insecurely on the frontend — for browser apps, prefer an httpOnly cookie over localStorage, which is exposed to XSS attacks.
  • Making tokens live too long — a stolen long-lived token is far more dangerous. Pair a short expiry with the refresh endpoint instead.
  • Mixing up guards — if auth:api isn't working, double-check config/auth.php actually points the api guard to the jwt driver.
  • Forgetting the Accept: application/json header — without it, Laravel may try to redirect an unauthenticated request to a login page instead of returning JSON.
12 · Wrap-up

What you've built

You now have a working Laravel API that can register users, log them in, protect routes so only valid tokens get through, refresh a token without asking for a password again, and log a user out. From here, natural next steps include role-based permissions (the spatie/laravel-permission package is a popular choice), rate-limiting your auth endpoints against brute-force attempts, and writing automated tests for each of these endpoints.

That's the full build.

Every piece — model, guard, controller, routes, middleware — works together to turn a signed string into a stateless proof of identity.
