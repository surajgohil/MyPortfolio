Laravel Eloquent Relationships Explained: A Beginner's Guide to hasOne, hasMany, belongsTo, and More

A plain-English walkthrough of Laravel's model relationships, with real examples you can copy into your own project today.

If you've worked with CodeIgniter or plain PHP before, you already know the pain of writing manual JOIN queries every time you need related data. Laravel's Eloquent ORM solves this with relationships — simple PHP methods that let you fetch connected data almost like magic.

In this guide, we'll break down the most important Eloquent relationships in plain English, with real examples you can copy into your own project today.

1. What Is an Eloquent Relationship, Really?

Think of your database tables as separate boxes of data. A relationship is just a rule that tells Laravel how two boxes connect — so instead of writing:

SELECT * FROM posts WHERE user_id = 5

You can just write:

$user->posts;

Laravel figures out the SQL behind the scenes. That's the whole point: less SQL, more expressive PHP.

2. The Foundation: Understanding Foreign Keys

Before diving into relationship methods, remember this simple rule that Laravel relies on:

The table that "belongs to" another table holds the foreign key.

Example: One User can have many Post records. So the posts table has a user_id column pointing back to the users table. Keep this picture in your head — it makes every relationship below click into place.

3. HasOne() — "I Have Exactly One"

Use hasOne() when one model owns exactly one related record.

Real-world example: A User has one Profile.

// User.php
class User extends Model
{
    public function profile()
    {
        return $this->hasOne(Profile::class);
    }
}

Database structure:

users table
id, name, email
profiles table
id, user_id, bio, avatar

Laravel automatically assumes the foreign key is user_id on the profiles table (based on the model name User). If your column is named differently, just tell Laravel:

return $this->hasOne(Profile::class, 'custom_user_id');

Usage:

$user = User::find(1);
echo $user->profile->bio; // Access it like a property, not a method!
💡 Beginner tip: Notice we wrote $user->profile, not $user->profile(). When you access it without parentheses, Eloquent runs the query and gives you the actual data. With parentheses, you get a query builder object (useful when you want to add more conditions).

4. HasMany() — "I Have Several"

Use hasMany() when one model can own multiple related records.

Real-world example: A User has many Post records.

// User.php
class User extends Model
{
    public function posts()
    {
        return $this->hasMany(Post::class);
    }
}

Database structure:

users table
id, name, email
posts table
id, user_id, title, body

Usage:

$user = User::find(1);

foreach ($user->posts as $post) {
    echo $post->title;
}

// Count them
echo $user->posts->count();

// Add extra conditions
$publishedPosts = $user->posts()->where('status', 'published')->get();

This is the relationship you'll use constantly — parent/child data like User → Orders, Category → Products, or Blog → Comments.

5. BelongsTo() — "I Belong to Someone Else"

belongsTo() is the inverse of hasOne() and hasMany(). It's defined on the model that actually holds the foreign key.

Real-world example: A Post belongs to a User.

// Post.php
class Post extends Model
{
    public function user()
    {
        return $this->belongsTo(User::class);
    }
}

Usage:

$post = Post::find(1);
echo $post->user->name; // Get the author's name

Quick Way to Remember the Direction

RelationshipDefined OnForeign Key Lives On
hasOneParent (User)Child table (profiles)
hasManyParent (User)Child table (posts)
belongsToChild (Post)Same table it's defined on

Every hasOne/hasMany relationship has a matching belongsTo on the other side. They're two halves of the same connection.

6. Putting It All Together: A Blog Example

class User extends Model
{
    public function posts()
    {
        return $this->hasMany(Post::class);
    }

    public function profile()
    {
        return $this->hasOne(Profile::class);
    }
}

class Post extends Model
{
    public function user()
    {
        return $this->belongsTo(User::class);
    }

    public function comments()
    {
        return $this->hasMany(Comment::class);
    }
}

class Comment extends Model
{
    public function post()
    {
        return $this->belongsTo(Post::class);
    }
}

Now you can chain relationships naturally:

$post = Post::find(1);
echo $post->user->name;           // Author's name
echo $post->comments->count();    // Number of comments

foreach ($post->comments as $comment) {
    echo $comment->post->title;   // Back to the post (inverse relationship!)
}

7. Bonus: The Other Relationship Types (So You Know They Exist)

Since you asked for the complete topic, here's a quick tour of the remaining relationship types you'll meet as you go further:

BelongsToMany() — Many-to-Many

Use when both sides can have multiple related records — like Post and Tag. Requires a pivot table (e.g., post_tag).

public function tags()
{
    return $this->belongsToMany(Tag::class);
}

HasManyThrough() — Access Data Through an Intermediate Model

Use when you want to reach a "grandchild" relationship. Example: A Country has many Posts through User.

public function posts()
{
    return $this->hasManyThrough(Post::class, User::class);
}

Polymorphic Relationships (morphTo / morphMany)

Use when one model can belong to more than one type of model. Example: Both Post and Video can have Comment records, using a single comments table.

// Comment.php
public function commentable()
{
    return $this->morphTo();
}

// Post.php and Video.php
public function comments()
{
    return $this->morphMany(Comment::class, 'commentable');
}

8. Common Beginner Mistakes to Avoid

  1. Forgetting parentheses confusion — $user->posts (property access, runs query) vs. $user->posts() (query builder, lets you chain ->where(), ->orderBy(), etc.)
  2. N+1 query problem — Looping through models and accessing a relationship inside the loop triggers a new query every time. Fix it with eager loading: 
    $users = User::with('posts')->get(); // loads all posts in one extra query
  3. Wrong foreign key naming — Eloquent guesses column names based on convention (user_id for a User model). If your migration used something custom, always pass it explicitly as the second argument.
  4. Not defining the inverse relationship — Always create both sides (hasMany on the parent AND belongsTo on the child) so you can navigate the relationship in both directions.

Final Thoughts

If you're coming from CodeIgniter where you'd normally write JOIN queries by hand, Eloquent relationships will feel like a superpower once they click. Start with hasOne, hasMany, and belongsTo — these three cover probably 80% of real-world use cases. Once you're comfortable, move on to belongsToMany for tags/categories-style features, and save polymorphic relationships for when you actually need one model to serve multiple parent types.

The best way to really learn this is to build something small: a blog with Users, Posts, and Comments is the perfect practice project, since it naturally uses all three core relationships covered here.
